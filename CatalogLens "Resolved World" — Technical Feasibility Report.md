# CatalogLens "Resolved World" — Technical Feasibility Report

## TL;DR
- **All four surfaces are feasible**, but the design hinges on one decision: IntelliJ's existing Gradle sync data (`ProjectDependencies`/`DependencyScopeNode`/`DependencyNode` under `ProjectKeys.DEPENDENCIES_GRAPH`) carries resolved coordinates + requested-by edges + a coarse conflict state only — it **flattens away** Gradle's structured `ComponentSelectionReason` descriptors, rejected-version lists, and exclude rules. Therefore the "Rejected & excluded" section and rich selection reasons **require CatalogLens to run its own Tooling API resolution via a custom `ToolingModelBuilder` injected by init script**, not to reuse sync data.
- **Recommended architecture:** inject an init script (`--init-script`) registering a custom `ToolingModelBuilder` that reads `configurations.X.incoming.resolutionResult` and serializes a typed, IDE-owned model (coordinates, `ComponentSelectionReason.getDescriptions()`, `isConstraint()`, `getFrom()`/requested-by edges) back over `org.gradle.tooling.GradleConnector`. Scope resolution to a single named classpath configuration to control cost.
- **Build order is sound:** (1) single-artifact "Why this version?" proves the Tooling-API plumbing end-to-end on one configuration; (2) full baseline snapshot + declarative inlay overlay; (3) the A→B delta with three sections. The declarative inlay API (`com.intellij.codeInsight.hints.declarative.InlayHintsProvider`) and `EditorNotificationProvider` are both current and suitable.

## Key Findings

### A. Getting a typed resolved graph from an IntelliJ plugin
- **Reusing IntelliJ sync data is insufficient for the full feature.** IntelliJ already resolves classpath configurations' `ResolutionResult` at sync time inside the Gradle daemon (via `ExtraModelBuilder` → `ModelBuilderService` implementations, fetched by `GradleModelFetchAction` using a `BuildActionExecuter`), and caches a graph as a `DataNode` of type `ProjectKeys.DEPENDENCIES_GRAPH` on each module. That graph is modeled as `ProjectDependencies` → `ComponentDependencies` (with `getCompileDependenciesGraph()` / `getRuntimeDependenciesGraph()`) → `DependencyScopeNode` → `DependencyNode` (package `com.intellij.openapi.externalSystem.model.project.dependencies`). It exposes resolved coordinate/display-name, child edges, and a coarse conflict/omitted state — enough for a tree and "requested by", but it does **not** carry structured `ComponentSelectionReason` causes, rejected candidate versions, or exclude-rule metadata. This data is free (already cached) and good for a first cut of the tree and the inlay overlay, but cannot power the "Rejected & excluded" section.
- **The robust approach is a custom Tooling model via init-script injection.** This is exactly the pattern IntelliJ itself uses (`GradleInitScriptUtil.createMainInitScript(...)` + `executionSettings.withArguments(GradleConstants.INIT_SCRIPT_CMD_OPTION, ...)`, i.e. `--init-script`). CatalogLens registers a `org.gradle.tooling.provider.model.ToolingModelBuilder` via `ToolingModelBuilderRegistry` in an injected init script; the builder reads `project.configurations.getByName(cfg).getIncoming().getResolutionResult()` and returns a serializable model. The model is requested from the plugin via `ProjectConnection.getModel(CustomModel.class)` or via a `BuildAction` executed with `ProjectConnection.action(...)`.
- **Tooling API entry point:** `org.gradle.tooling.GradleConnector.newConnector().forProjectDirectory(...).connect()` → `ProjectConnection`. From a plugin you should reuse IntelliJ's plumbing — `GradleExecutionHelper.execute(...)` with a `GradleExecutionContext` (the older `getBuildLauncher`/`getBuildEnvironment(ProjectConnection,...)` overloads are now `@Deprecated`) and `GradleExecutionSettings` obtained via `ExternalSystemApiUtil.getExecutionSettings(project, projectPath, GradleConstants.SYSTEM_ID)`. Critically, IntelliJ's `GradleExecutionHelper.execute()` historically did **not** inject your model builder's init script automatically; you must add your `--init-script` argument yourself (this was the root cause in the known "No builders are available to build a model of type" support thread). The alternative IDE-native route is a `projectResolve` extension (`AbstractProjectResolverExtension`/`populateModuleExtraModels`) but that ties extraction to sync and is heavier than an on-demand connection for an interactive feature.
- **BuildAction route:** `ProjectConnection.action(BuildAction<T>)` returns a `BuildActionExecuter<T>`; the action is serialized into the daemon and runs against a `BuildController` (which can fetch per-project models). This lets you read `ResolutionResult` for a chosen project/configuration in-process and serialize a compact typed result back, avoiding stdout parsing entirely. (Historically `action()` required Gradle 1.8+ and phased actions 4.8+, but note Gradle's current `GradleConnector` javadoc, API 9.5.0, only guarantees Tooling API support for "Gradle version 4.0 and up," which is the window that matters for your 8.x/9.x targets.)
- **Most efficient/robust:** target one resolvable configuration at a time; the `ResolutionResult` graph is the *graph-resolution* phase only (`getResolutionResult()` "will resolve the dependency graph but will not resolve or download the artifacts"), which is cheaper than artifact resolution. The Tooling API is version-independent: per Gradle's official Tooling API user guide, "The Tooling API version is guaranteed to support running builds with all Gradle versions for the last five major releases. For example, the Tooling API 9.0.0 release is compatible with Gradle…", and `GradleConnector` (API 9.5.0) confirms support for "running builds using Gradle version 4.0 and up." This comfortably covers the Gradle 8.x/9.x + AGP 8.x targets.

### B. Selection reasons / rejected / excluded data (the make-or-break for section 3c)
- **`ResolvedComponentResult.getSelectionReason()`** returns a **`ComponentSelectionReason`**, whose `getDescriptions()` returns `List<ComponentSelectionDescriptor>`; each descriptor exposes a **`ComponentSelectionCause`** plus a human-readable description. Per Gradle's official `ComponentSelectionCause` javadoc, the enum values are: `REQUESTED`, `SELECTED_BY_RULE`, `FORCED`, `CONFLICT_RESOLUTION`, `COMPOSITE_BUILD`, `REJECTION`, `CONSTRAINT`, and `BY_ANCESTOR` ("This component was selected because it was requested by a parent with a strict version"). `ComponentSelectionReason` also has boolean helpers: `isConflictResolution()`, `isConstrained()`, `isForced()`, `isSelectedByRule()`, `isExpected()`, `isCompositeSubstitution()`. (Note: the old single `getDescription()` was removed in favor of `getDescriptions()`.)
- **"Requested by" / requester edges:** `DependencyResult.getFrom()` returns the origin `ResolvedComponentResult`; `getRequested()` returns the `ComponentSelector` (which may be a `ModuleComponentSelector` or `ProjectComponentSelector`); `ResolvedDependencyResult.getSelected()` returns the chosen component. `ResolvedComponentResult.getDependents()` gives incoming edges directly. This fully supports the expandable "requested by" subtree.
- **Constraints:** `DependencyResult.isConstraint()` indicates the edge originated from a dependency constraint; combined with `ComponentSelectionCause.CONSTRAINT` this surfaces rich-version/constraint-driven selection.
- **Rejected versions (rich-version `reject()` / component-selection `reject()`):** rejection surfaces as `ComponentSelectionCause.REJECTION` in the selection reason of an *unselected* candidate, and rejected dynamic candidates appear via `UnresolvedDependencyResult` (`getAttempted()`/`getFailure()`). This is the PRIORITY data for section 3c and is reachable, but only from a full `ResolutionResult` you compute yourself — IntelliJ's cached graph does not retain it.
- **Excludes:** configuration-level excludes (`Configuration.exclude(group/module)`) and per-dependency excludes (`ModuleDependency.getExcludeRules()`) are **declaration-side metadata**, not first-class fields of the resolved graph. The *effect* (a transitive that never appears) is visible by absence; to label something "excluded via exclude rule" you must read the exclude rules off the `Configuration`/`Dependency` objects in your model builder and correlate. So section 3c's secondary "excluded" data requires reading exclude rules in the builder, not just the graph.

### C. Configuration / variant enumeration
- For an Android module, AGP defines per-variant resolvable classpaths: `debugCompileClasspath`, `debugRuntimeClasspath`, `releaseCompileClasspath`, `releaseRuntimeClasspath`, plus test variants (`testDebugRuntimeClasspath`, `androidTestDebugRuntimeClasspath`, etc.). These are the "interesting" ones for a resolved-world view.
- Enumerate via `project.getConfigurations()` and filter with `Configuration.isCanBeResolved()` (resolvable, not declarable/consumable). Then whitelist by name suffix (`*RuntimeClasspath` / `*CompileClasspath`) to cut from the hundreds of internal configurations (the `*ApiElements`, `*RuntimeElements`, metadata, lint, etc.) down to the handful users care about. The dependencyInsight precedent confirms `--configuration debugRuntimeClasspath --dependency <coords>` as the canonical scope.

### D. Cost / performance
- Resolving a single classpath configuration's `ResolutionResult` triggers only the graph-resolution phase (no artifact download), and can be scoped to exactly one `(module, configuration)` — this is the key lever to keep an interactive re-resolve fast. Avoid `resolutionResult.getAllComponents()` across all configurations.
- Reusing IntelliJ's sync graph is effectively free (already computed at sync) and is the right source for the inlay overlay and the initial tree, but it cannot supply selection reasons/rejects/excludes, so the delta's section 3c and "Why this version?" still need an on-demand resolve.
- On large multi-module Android projects, the first connection warms a Gradle daemon; subsequent resolves reuse it. Cache the serialized typed model per `(module, configuration, catalog-hash)` so the baseline survives until the catalog actually changes. Configuration-cache interactions are a known hazard (the gradle-versions-plugin documents that cross-project configuration resolution and `configuration-cache.problems=warn` can break dependency reads in Gradle 9+); prefer resolving the target project's own configuration in-process.

### E. Declarative inlay hints
- Use **`com.intellij.codeInsight.hints.declarative.InlayHintsProvider`**, registered in the **`com.intellij.codeInsight.declarativeInlayProvider`** extension point. The legacy `InlayHintsProvider` class KDoc in JetBrains/intellij-community (`platform/lang-api/.../hints/InlayHintsProvider.kt`) explicitly steers you to the declarative API: *"ATTENTION! Consider using [com.intellij.codeInsight.hints.declarative.InlayHintsProvider] whenever possible! It is order of magnitude faster, much simpler and less error-prone. This class is very likely to be deprecated in the future."*
- End-of-line placement is supported: collectors add presentations at an `EndOfLinePosition(lineNumber)`, and presentations are built with a small DSL (`text(...)`, plus `InlayActionPayload`/`StringInlayActionPayload` for clickable items) — confirmed by a working community example placing end-of-line hints. Click handlers are supported via inlay action payloads + a registered `InlayActionHandler`.
- For a non-Java language like TOML, register a `com.intellij.codeInsight.hints.declarative.InlayHintsProviderFactory` (EP `com.intellij.codeInsight.declarativeInlayProviderFactory`) returning `InlayProviderInfo` if the per-language registration is awkward; otherwise register the provider with the TOML language id. Attaching hints to specific `[libraries]`/`[plugins]` lines works by mapping each catalog entry's PSI/line to an end-of-line presentation, shown only when resolved ≠ declared.

### F. Editor notification banner
- Use **`com.intellij.ui.EditorNotificationProvider`** (current form), registered in the **`com.intellij.editorNotificationProvider`** extension point. Implement `collectNotificationData(project, file)` returning a function `FileEditor -> JComponent` (the convenience base `EditorNotificationProvider.Provider<T>` delegates to `createNotificationPanel(file, fileEditor, project)`). Use `EditorNotificationPanel` as the UI; mark the provider `DumbAware` if it needs no indexes. Trigger refresh with `EditorNotifications.getInstance(project).updateNotifications(file)` when the baseline goes stale. (`EditorNotificationProvider`, singular, is the modern replacement for the older `EditorNotifications.Provider`.)

### G. Mapping resolved graph back to the catalog
- Feasible. CatalogLens already parses `libs.versions.toml`, so build a map from `group:name` → catalog alias/line, resolving `version.ref` indirection during parse. Resolved components expose `ModuleComponentIdentifier.getGroup()/getModule()/getVersion()`; match on `group:name` (ignore version). Three cases: (1) catalog-declared library → map to its `[libraries]` line; (2) version.ref plugins/libraries → resolve the ref to the `[versions]` entry; (3) transitive-only artifacts not in the catalog → flag as "transitive, not in catalog" with their requested-by chain. Because aliases can collide on subgroup accessors (e.g. `jackson` vs `jackson.databind`), key on `group:name`, not on alias text.

### H. Tool window / tree-table UI
- `com.intellij.openapi.wm.ToolWindowFactory` registered with `anchor="bottom"` gives the bottom-docked window. For the three-section delta, use `com.intellij.ui.treeStructure.treetable.TreeTable` with a `ListTreeTableModel` and `ColumnInfo[]` columns (e.g., Coordinate, Declared, Resolved, Reason). Use the newer **`JBTreeTable`** (it correctly shows expand/collapse handles and supports `setColumnProportion(...)`; plain `TreeTable` is fussier about root handles). Add `TreeTableSpeedSearch` for filtering. Diff coloring can pull semantic colors from the active `EditorColorsScheme` (added/removed/changed) so the delta matches the user's theme.

### I. File-change detection (cheap, no resolution)
- Detect catalog edits without resolving via any of: a VFS listener (`BulkFileListener` on `VirtualFileManager`/`VFS_CHANGES`), a document listener on the catalog's `Document`, or a `PsiModificationTracker`/`ModificationTracker` count compared to the snapshot. Cheapest is to hash the catalog text (or store the PSI modification stamp) at baseline and compare on edit; only flip the EditorNotification banner to "stale" — never auto-resolve. Re-resolve happens only on explicit user action ("set new baseline" / "compute delta").

### J. Prior art
- **IntelliJ's own Gradle Dependency Analyzer** (shipped in IntelliJ IDEA 2022.1, published April 11, 2022) and the Gradle tool window dependencies node: show resolved tree, conflicts (greyed/"omitted"), unresolved, and transitive parentage — but expose conflict only as a binary state, not Gradle's textual reasons/rejects/excludes. This confirms what's possible from cached sync data and what is missing (the gap CatalogLens fills).
- **`rholder/gradle-view`** — an IntelliJ plugin that "rolls up the entire dependency graph into a pair of digestible trees derived from a serialized version of Gradle's dependency graph," built on the Gradle Tooling API. Direct precedent for serializing the graph over TAPI and rendering trees; greys out already-included dependencies (the same omitted-for-conflict signal).
- **`ben-manes/gradle-versions-plugin`** — applies via init script across all projects and uses `resolutionStrategy.componentSelection { ... reject(...) }`; documents `filterConfigurations { it.name == "runtimeClasspath" || it.name == "compileClasspath" }` and the Gradle 9 parallel/configuration-cache caveats — directly relevant to scoping and pitfalls.
- **`autonomousapps/dependency-analysis-gradle-plugin`** — analyzes resolved graphs per source set/variant, prints a dominator tree and a `reason` task; good model for variant-aware analysis and "why" explanations, though it's a build plugin, not an IDE feature.
- **`dependencyInsight`** is the semantic target for surface 4: `./gradlew :app:dependencyInsight --configuration debugRuntimeClasspath --dependency <coords>` produces exactly the "Selection reasons: By constraint… By conflict resolution: between versions …" output you want to reproduce in-IDE from `ComponentSelectionReason`.

## Details

### Recommended data-extraction architecture
Run an **on-demand Tooling API connection** that injects an init script registering a CatalogLens `ToolingModelBuilder`. The builder, given a `(project, configurationName)`:
1. `Configuration cfg = project.configurations.getByName(name)` — guard with `cfg.isCanBeResolved()`.
2. `ResolutionResult rr = cfg.incoming.resolutionResult`.
3. Walk `rr.root` via `getDependencies()`; for each `ResolvedDependencyResult` capture `getSelected()` (coordinates + `getSelectionReason().getDescriptions()` mapped to your own enum mirroring `ComponentSelectionCause`), `getRequested()`, `isConstraint()`, and `getFrom()` for the requested-by edge. Capture `UnresolvedDependencyResult`s for rejected/failed candidates.
4. Read `cfg.excludeRules` and per-`ModuleDependency` `getExcludeRules()` for the excluded set.
5. Serialize a compact POJO model (interfaces with getters only — TAPI proxies them) back to the plugin.

This keeps everything **typed**, avoids stdout parsing, and is the only way to get reasons/rejects/excludes. Use IntelliJ's `GradleExecutionHelper` + `GradleExecutionSettings` so the connection reuses the project's Gradle distribution, JVM args, and daemon.

### Version constraints
- Tooling API: ≥ Gradle 4.0 supported by TAPI 9.x; the selection-reason descriptors API (`getDescriptions()`) is stable across 8.x/9.x. (`BuildAction`/`action()` and phased actions predate the current supported window — don't rely on pre-4.0 behavior.)
- IntelliJ Platform: declarative inlay API (`com.intellij.codeInsight.hints.declarative.*`) and `EditorNotificationProvider` are current in 2024.x–2025.1+ builds.
- Android Studio: variant classpath configuration names (`{variant}RuntimeClasspath`/`{variant}CompileClasspath`) are AGP 8.x conventions; resolving them may require providing the AGP attributes (build type/agp version/usage) the way dependencyInsight does, or simply resolving the named configuration in-project where attributes are already set.

## Recommendations

**Stage 1 — Prove the plumbing with "Why this version?" (surface 4).**
- Build the injected-init-script `ToolingModelBuilder` returning a single-artifact insight model for one `(module, configuration, coordinates)`: selected version, `ComponentSelectionReason` descriptions, the requested-by chain (`getFrom()` walk). Reach it from an editor action on a `[libraries]` entry.
- Benchmark a cold and warm resolve on a representative large multi-module Android project. **Threshold to proceed:** warm single-configuration resolve under ~2–3 s; if not, add per-`(module,configuration,catalog-hash)` caching before Stage 2.
- Decision gate: if `ComponentSelectionReason.getDescriptions()` reliably yields CONSTRAINT/CONFLICT_RESOLUTION/REJECTION causes on your target Gradle/AGP, proceed; if rejects don't surface as expected, design section 3c as "best-effort" with excludes read from configuration rules.

**Stage 2 — Baseline snapshot + resolved-vs-declared inlay overlay (surface 3).**
- Extend the model builder to return the full resolved graph for the chosen configuration; snapshot it as the baseline keyed by catalog hash.
- Implement the declarative `InlayHintsProvider` (+ factory for TOML) showing end-of-line `resolved X (declared Y)` only when they differ; map via `group:name`.
- Add the `EditorNotificationProvider` "resolved data is stale" banner, flipped by the cheap catalog-hash/PSI-stamp comparison. For the first cut you may read the tree from IntelliJ's cached `ProjectKeys.DEPENDENCIES_GRAPH` to render fast, upgrading to your own model where reasons are needed.

**Stage 3 — A→B delta with three sections (surfaces 1 & 2).**
- Diff baseline vs. re-resolved graph. Classify: (a) "You changed" = component deltas whose requested edge traces to a catalog line you edited; (b) "Transitive ripple" = bumped/downgraded/added/removed with expandable requested-by subtrees + reasons; (c) "Rejected & excluded" = REJECTION-cause candidates and exclude-rule victims (priority on rejects, excludes secondary).
- Render in a bottom `ToolWindowFactory` using `JBTreeTable`/`ListTreeTableModel` with theme-aware diff colors from `EditorColorsScheme`.
- **Benchmark to revisit:** if re-resolve on edit feels slow, scope to the single active variant configuration and resolve lazily on "compute delta" only.

**Cross-cutting:** never auto-resolve on keystroke; gate all resolution behind explicit baseline/delta actions and the stale banner. Guard for configuration-cache/parallel pitfalls by resolving the target project's own configuration.

## Caveats
- IntelliJ's cached sync graph (`DependencyNode`) is convenient but **lossy** for reasons/rejects/excludes; the exact `getResolutionState()`/`getSelectionReason()` signatures on `AbstractDependencyNode` and the Gradle-side builder that maps `ResolutionResult`→`ProjectDependencies` still need final source confirmation against `platform/external-system-api/.../model/project/dependencies/`. Treat the cached graph as render-only and your own TAPI model as authoritative.
- Exclude attribution is inherently partial: an excluded transitive is invisible in the resolved graph, so "excluded" is reconstructed from declaration-side exclude rules, not observed graph state — label it as such.
- AGP variant configurations can fail to resolve standalone if requested without the right attributes; resolving the named configuration within its own project context (as the model builder does) avoids most of this, but verify on a real AGP 8.x project.
- Gradle 9 + configuration cache / parallel execution can interfere with on-demand resolution (documented by gradle-versions-plugin); validate against Gradle 9.x early.
- The declarative inlay API is marked experimental and "may change without preserving backward compatibility"; pin to the platform version you build against and re-test on upgrades.
- Tooling API version compatibility: TAPI guarantees support for the last five major Gradle releases and the current+next major going forward — fine for 8.x/9.x today but worth a compatibility check when Gradle 10 lands.
